# /ingest-policy

Generate the institution-specific rulebook (`.claude/rules/policy.md`) from a reimbursement-policy document. **This is what makes the administrative workflow portable across institutions** — instead of hand-transcribing rules, drop in your policy and let the parser build the rulebook in the exact schema `compliance-reviewer` already enforces.

## Instructions

1. **Locate the policy document.** Use `$ARGUMENTS` if given; otherwise look in `policy_source/` or the project root for a likely file (`*polic*`, `*reimburs*`, `*expense*`, a `.pdf`/`.docx`). If none or ambiguous, ask the user to drop the document in and provide the path.

2. **Extract the text.**
   - **PDF** → invoke the `pdf` skill. If the PDF is scanned (no text layer), use its OCR mode and **flag that OCR was used** so the user can sanity-check.
   - **Word** → invoke the `docx` skill.
   - **`.md` / `.txt`** → read directly.

3. **Invoke the `policy-parser` agent** with the extracted text. It returns: the proposed `policy.md` body (rules grouped by category, each with an ID, severity, and a source citation), plus a *Confirmations needed* list, a *Severity judgment calls* list, and a *Coverage note*.

4. **Review with the user BEFORE writing.** Surface:
   - the proposed category sections and rule count per section,
   - every `[NEEDS CONFIRMATION]` item (resolve each with the user),
   - the severity judgment calls,
   - the coverage gaps (schema sections the document didn't populate).
   Do **not** write `policy.md` until the user signs off.

5. **Write the rulebook** to `.claude/rules/policy.md`, with a header line:
   `> Generated by /ingest-policy from <doc> on <date>. Re-run to refresh.`
   Keep each rule's source citation. Leave any still-unresolved item as an explicit `[NEEDS CONFIRMATION]` in the file (so it's visible at review time).
   - **Re-ingestion** (policy.md already exists, e.g. a new fiscal year / revised caps): show a **diff** of what changed, confirm, and keep the prior file as a dated backup (`policy.<YYYY-MM-DD>.bak.md`) before overwriting.

6. **Offer the companion updates** (only after the rulebook is approved):
   - Update `CLAUDE.md`'s **Governing policy** line and the **Non-negotiables** summary to match the new rulebook.
   - If the document specifies form fields / budget codes, propose filling the **Form Defaults** table in `CLAUDE.md` and mapping cells in `agents/form-filler.md`.

7. **Report:** sections created, total rules, count of unresolved `[NEEDS CONFIRMATION]` items (these limit what `compliance-reviewer` can fully check until resolved), and the file path.

## Notes

- **Rigor first.** Never let the parser invent a rule. An unresolved `[NEEDS CONFIRMATION]` is safer than a fabricated ceiling — `compliance-reviewer` cites these rule IDs, so an error here propagates into every audit.
- **Portability.** The schema (rule IDs + severity + exceptions) is universal; only the *content* is institution-specific. Moving to a new institution = drop the new policy doc, re-run `/ingest-policy`.
- **Multi-language.** The parser handles non-English policies; the rulebook is written in English with original terms preserved.

## Examples

### Example 1 — first setup at a new institution
**User:** `/ingest-policy policy_source/Travel_and_Expense_Policy_2026.pdf`
**Actions:** extract via `pdf` skill → `policy-parser` produces sections A (universal: 60-day deadline, originals required), B (travel: economy only, book via agency, waivers listed), D (hotel ceilings by city), E (meals €/day), F (IT via procurement), exclusions → 3 `[NEEDS CONFIRMATION]` (per-diem rate ambiguous; "incidental" undefined; software cap not stated) → review with user → write `rules/policy.md` → offer to update CLAUDE.md non-negotiables. **Result:** institution rulebook generated; 3 items flagged for the user.

### Example 2 — policy revised
**User:** "They updated the hotel ceilings — re-ingest."
**Actions:** locate the new doc, parse, **diff** against the current `policy.md` (only section D changed), confirm, back up the old file, write the update.

## Troubleshooting

- **Scanned PDF, garbled extract** → re-run the `pdf` skill in OCR mode; if still poor, ask for a text/Word version.
- **One document covers several policies** (travel vs. general expenses) → ingest each, or ask the user which sections are in scope.
- **Document is silent on a whole category** → leave that schema section as a stub with `[NEEDS CONFIRMATION: not addressed in <doc>]`; don't fabricate it.
